# Lab: Beginner Best Practices For GitHub Copilot Agents

## Scenario Context
Your BI, DS, or DE team is preparing to use GitHub Copilot agents in VS Code for the first time.
The goal is to practice the core beginner guardrails before moving into setup and implementation work.

## Learner Goal
By the end of this lab, you should be able to:
- identify the safest starting mode for a task
- improve a vague prompt into a specific one
- explain why review is required before accepting output
- name one team rule that should be standardized

## Starter State
You begin with:
- VS Code open
- Copilot Chat available or demonstrable in the environment
- a small practice workspace or sample file

## Target State
You finish with:
- one improved prompt
- one explanation of when to use Ask, Plan, and Agent
- one review rule and one team guardrail written down

## Prerequisites And Setup
Goal of the step:
- Make sure the learner can focus on behavior and guardrails rather than environment setup details.

Exact actions:
- Open a safe practice workspace.
- Open Chat if available.
- Choose one small file or folder to reference.

What success looks like:
- You can point to the sample context you will use in the exercise.

Common failure mode and fix:
- Failure: The workspace is too large or unclear for a first exercise.
- Fix: Use a small README or support file instead.

## Lab Tasks

### Task 1. Choose The Safest Mode
Goal of the step:
- Match Ask, Plan, and Agent to the right job.

Exact action:
- For one explanation task, one planning task, and one small edit task, decide which mode is safest.

What success looks like:
- You can justify your mode choice.

Common failure mode and fix:
- Failure: You choose Agent for everything.
- Fix: Start with the lightest mode that can answer the need.

### Task 2. Improve A Weak Prompt
Goal of the step:
- Turn a broad request into a specific prompt.

Exact action:
- Rewrite a vague prompt such as `make this better` into a prompt that includes goal, scope, output, and validation.

What success looks like:
- The new prompt is specific enough to review objectively.

Common failure mode and fix:
- Failure: The prompt sounds better but still lacks boundaries.
- Fix: Add file scope and expected output.

### Task 3. Define One Review Rule
Goal of the step:
- Build a review habit before using AI output.

Exact action:
- Write one rule you will follow before accepting any Copilot output.

What success looks like:
- The rule is concrete, such as reviewing the diff or checking business meaning.

Common failure mode and fix:
- Failure: The rule is too vague.
- Fix: Tie it to an action you can observe.

### Task 4. Define One Team Guardrail
Goal of the step:
- Identify one team behavior that should become standard.

Exact action:
- Write one reusable guardrail such as `Always start multi-step work with Plan` or `Never accept a change you cannot explain.`

What success looks like:
- The guardrail can apply across multiple tasks.

Common failure mode and fix:
- Failure: The guardrail applies only to one file.
- Fix: Rewrite it as a general team rule.

## Suggested Learner Prompts
- Explain the difference between Ask, Plan, and Agent for a new data team member.
- Review this prompt and make it safer for a beginner: `make this notebook better`.
- Suggest one review rule for a BI, DS, or DE team using Copilot for the first time.
- Before making changes, ask clarifying questions if the request is ambiguous.

## Expected Observable Output
- One improved prompt
- One mode-selection explanation
- One written review rule
- One written team guardrail

## Validation Checklist
- I can explain when Ask, Plan, and Agent should be used.
- I can improve a vague prompt into a safer one.
- I can state one concrete review rule.
- I can state one reusable team guardrail.

## Reflection Tasks

- Which guardrail should be mandatory from day one?
- What makes a prompt safe enough to run?
